- Runtime identity and env resolution are centralized in `apps/server/src/cli.ts`, `apps/server/src/os-jank.ts`, and `scripts/dev-runner.ts`.

## Resolved Issues

- 2026-04-06: Fixed daemon-status stale metadata drift found during a Spec Compliance review sweep. The CLI status snapshot still surfaced trusted `daemon.json` PID/WebSocket/start-time fields even after `daemon.ping` failed, so `forge daemon status` could report a stopped daemon alongside stale live-runtime metadata after a crash or partial teardown. The fix suppresses manifest-derived live metadata unless the socket is actually responding, adds focused CLI client coverage for stopped snapshots with stale manifests, adds status-render regression coverage for the stopped output, and verifies the change with `bun fmt`, `bun lint`, `bun typecheck`, and `bun run test`.
- 2026-04-06: Fixed a daemon socket trust gap found during a Security review sweep. Desktop discovery, CLI RPC, and daemon singleton probing still followed or trusted `forge.sock` paths without verifying that the socket was the real owner-only socket Forge writes, so a symlinked or overly broad socket file under `~/.forge` could be treated as a healthy daemon. The fix adds shared trusted-socket helpers that reject symlinked or non-`0600` socket paths, wires those checks into desktop/CLI/daemon socket probes, updates the daemon/CLI harnesses to create spec-compliant sockets, and verifies the change with focused shared, desktop, CLI, and daemon-service regression coverage plus `bun fmt`, `bun lint`, `bun typecheck`, and `bun run test`.
- 2026-04-06: Fixed a singleton socket-reuse regression found during a Spec Compliance review sweep. `DaemonService.start()` still treated a responsive `forge.sock` as reusable only when it could also recover a PID from `daemon.ping`, `forge.pid`, or `daemon.json`, which violated the daemon lifecycle spec that a successful ping alone means the singleton is already running. The fix makes socket responsiveness authoritative even without recoverable PID state, widens the internal already-running result to tolerate an unknown PID, and adds focused daemon-service regression coverage for a pid-less `daemon.ping` response after both state files are missing.
- 2026-04-06: Fixed daemon session-create explicit-type drift found during a Spec Compliance review sweep. The daemon transport still accepted a `session.create` `type` field but ignored it, and the CLI lacked the documented `forge create --type ...` path. The fix adds `forge create --type agent|workflow`, validates explicit create types in the socket transport instead of silently dropping them, and adds focused CLI/transport regression coverage for valid workflow creates plus mismatched type rejection.
- 2026-04-06: Fixed a daemon state-file symlink hardening gap found during a Security review sweep. Desktop/CLI manifest discovery and daemon startup still followed symlinked `daemon.json` / `forge.pid` paths, which could trust or overwrite arbitrary user-owned files under `~/.forge`. The fix adds shared no-follow daemon-manifest readers, makes daemon state-file reads/writes open with `O_NOFOLLOW` where supported, and adds regression coverage for symlinked manifest/state paths across shared, desktop, CLI, and daemon-service tests.
- 2026-04-06: Fixed an observability documentation identity leak found during a Product Identity review sweep. `docs/observability.md` still documented the retired `t3_*` metric families even though runtime metrics now emit `forge_*`, which left the published troubleshooting guidance out of sync with the actual Forge telemetry surface. The fix renames the documented metric families to `forge_*` and adds a regression test that keeps the observability doc aligned with the Forge metric namespace.
- 2026-04-06: Fixed a remaining observability identity leak found during a Product Identity review sweep. The live server metrics still published under `t3_*` names, which leaked the fork origin into emitted telemetry even after the Forge cutover. The fix renames the runtime metric namespace to `forge_*`, updates the affected metric assertions, and adds a product-identity regression guard against reintroducing `t3_` prefixes in non-test server source.
- 2026-04-06: Fixed a daemon socket memory-pressure gap found during a Security review sweep. The Unix socket transport buffered inbound bytes until newline with no request-size cap, so a misbehaving local client could grow per-connection memory without bound before JSON-RPC parsing ran. The fix adds a 1 MiB request-line ceiling, returns a JSON-RPC invalid-request error when the limit is exceeded, closes the offending connection, and adds focused transport coverage for the oversized-request path.
- 2026-04-06: Fixed a daemon JSON-RPC correlation gap found during a Security review sweep. The desktop daemon probe and CLI socket client accepted the first successful-looking JSON line from the Unix socket without verifying that the JSON-RPC response `id` matched the request they sent, which could treat stale or misrouted socket traffic as a healthy Forge daemon reply. The fix validates the JSON-RPC envelope/id on both paths and adds focused regression coverage for mismatched response ids.
- 2026-04-06: Fixed a daemon event-subscription cursor compatibility gap found during a Spec Compliance review sweep. The daemon docs/contracts describe `events.subscribe` reconnect cursors as `fromSequence` / `fromSequenceExclusive`, but the Unix socket transport only accepted the internal `afterSequence` field. The fix normalizes all three cursor aliases, rejects conflicting cursor params, and adds focused socket transport coverage so documented daemon clients can reconnect without transport-specific field drift.
- 2026-04-06: Fixed a daemon event-stream cursor bug found during a Spec Compliance review sweep. `events.subscribe` returned `nextSequenceExclusive` equal to the last delivered sequence instead of the next unread sequence, so `forge events` could replay the most recent event forever when it advanced its cursor. The fix makes the socket transport return a truly exclusive cursor and updates the daemon transport/CLI regression coverage to lock the contract in place.
- 2026-04-06: Fixed daemon-stop CLI completion drift found during a Spec Compliance review sweep. `forge daemon stop` returned as soon as the JSON-RPC stop request was queued, so follow-up automation could race a daemon that still owned `forge.sock` and had not finished graceful shutdown. The fix makes the CLI wait for the daemon to stop responding before returning, reuses the same stop helper for `forge daemon restart`, and adds regression coverage for the wait path.
- 2026-04-06: Fixed a first-run daemon startup bug found during a Spec Compliance review sweep. `DaemonService.start()` tried to acquire the `forge.lock` flock/lockf helper before creating the Forge base directory, so daemon startup could fail on a clean machine with no existing `~/.forge`. The fix creates the lock path parent directory before lock acquisition and adds regression coverage for first-run startup against a missing base dir.
- 2026-04-06: Fixed a daemon session classification/routing bug found during the Spec Compliance review sweep. Top-level direct agent sessions were still being surfaced as `chat` in the daemon socket transport and WebSocket projections, and `session.correct` was always routed as guidance instead of sending a normal follow-up turn for standalone agent sessions. The fix centralizes session-type derivation, restores top-level direct sessions to `agent`, routes standalone-agent `session.correct` through `thread.send-turn`, and adds focused regression coverage for both the listing and correction paths.
- 2026-04-06: Fixed a daemon socket version-skew gap found during the Spec Compliance review sweep. The Unix socket transport accepted CLI JSON-RPC requests without any protocol handshake, so newer or older `forge` binaries could talk to a mismatched daemon with no upgrade/restart guidance. The fix adds an explicit daemon socket protocol version contract, has the CLI attach its protocol version to each socket RPC, rejects missing or mismatched versions server-side for non-ping methods with `forge daemon restart` instructions, and adds regression coverage for both the client metadata and transport-side mismatch handling.
- 2026-04-06: Fixed a daemon reconnect-token contract gap found during a Security review sweep. Shared `daemon.json` parsing still accepted any non-empty `wsToken`, and daemon-mode config resolution would honor arbitrary inherited `FORGE_AUTH_TOKEN` overrides, which could persist malformed or non-rotating reconnect tokens that desktop and CLI should never trust. The fix tightens shared manifest parsing to require the documented 256-bit hex token format, makes daemon mode always generate a fresh startup auth token instead of reusing caller-provided overrides, updates the affected daemon/desktop/shared fixtures, and adds regression coverage for invalid token manifests plus daemon-mode token rotation.
- 2026-04-06: Fixed a detached-daemon bootstrap/env leak found during a Security review sweep. Both daemon launch paths still allowed one-shot runtime overrides like `FORGE_BOOTSTRAP_FD` to bleed into spawned daemon processes, which could force daemon mode to consume stale bootstrap descriptors or other inherited runtime state. The fix extracts shared daemon child-env sanitization into `@forgetools/shared/daemon`, applies it to both CLI and desktop detached launches, and adds regression coverage for the stripped env keys.
- 2026-04-06: Fixed a daemon auth-token rotation leak found during a Security review sweep. The CLI daemon launcher inherited `FORGE_AUTH_TOKEN` from the parent environment, which could pin daemon-mode WebSocket auth across restarts instead of generating the required fresh token per startup. The fix strips inherited `FORGE_AUTH_TOKEN` alongside the other daemon runtime overrides in the launch plan and adds regression coverage for the detached-launch environment.
- 2026-04-06: Fixed a follow-up desktop daemon discovery race found during the Integration review sweep. `ensureDaemonConnection()` still fell through to `spawnDetachedDaemon()` after the manifest wait deadline if an existing daemon kept answering `daemon.ping` but never republished trusted `daemon.json` state, causing a pointless duplicate-launch attempt right before surfacing the manifest-missing error. The fix tracks when desktop actually waited on a responsive socket, refuses the detached relaunch in that state, and adds regression coverage for the live-daemon-without-manifest timeout path.
- 2026-04-06: Fixed a residual checkpoint identity leak found during the Product Identity review sweep. Forge was still creating live git checkpoint refs under `refs/t3/checkpoints/...`, which leaked the old product name into repository state even after the broader identity cutover. The fix moves generated checkpoint refs to `refs/forge/checkpoints/...`, updates the affected orchestration projection fixtures, and adds a focused regression test covering the Forge checkpoint namespace.
- 2026-04-06: Fixed bootstrap recovery CLI drift found during the Spec Compliance review sweep. The daemon-mode design documents `forge bootstrap-retry` and `forge bootstrap-skip`, but the CLI and socket transport still forced operators to resolve bootstrap-failed requests manually by request id. The fix adds daemon-side bootstrap request resolution by session id, wires the missing CLI commands, and adds transport/CLI regression coverage for the new recovery path.
- 2026-04-06: Fixed daemon monitoring CLI drift found during the Spec Compliance review sweep. The daemon design still promised `forge watch`, `forge logs`, `forge events`, and an `events.subscribe` socket path, but the implemented daemon surface stopped at control commands. The fix adds a long-poll `events.subscribe` JSON-RPC method that safely merges replay plus live orchestration events, wires the three monitoring CLI commands onto the daemon socket/transcript APIs, and adds focused transport plus CLI regression coverage.
- 2026-04-06: Fixed a desktop daemon discovery race found during the Integration review sweep. `ensureDaemonConnection()` would respawn a detached daemon whenever `daemon.json` was briefly missing, even if the existing Forge socket was already responsive, creating needless duplicate-launch attempts during manifest write races. The fix treats socket responsiveness as authoritative, waits for trusted manifest state before relaunch, surfaces a clearer error if a live daemon never republishes `daemon.json`, and adds desktop regression coverage for the live-daemon-without-manifest path.
- 2026-04-06: Fixed a desktop daemon-manifest cache drift found during the Integration review sweep. The preload-side WebSocket URL resolver treated the first trusted `daemon.json` read as permanent state, so a daemon restart that rotated `wsPort`/`wsToken` could leave subsequent desktop URL lookups pinned to stale connection info until a full app restart. The fix makes the resolver re-read trusted manifest state on `getWsUrl()`/`prime()`, preserves the last known-good URL only as a fallback when the manifest is briefly unavailable, and adds regression coverage for manifest rotation and repeated priming.
- 2026-04-06: Fixed a residual product identity leak found during a follow-up Product Identity review sweep. Production brand asset filenames and the shared packaging asset map still used `t3-black-*`, leaving old T3 naming in shipped web/favicon and Windows icon build inputs. The fix renames those production assets to `forge-black-*`, updates the shared brand asset manifest, and restores a clean grep for packaged T3-era asset names.
- 2026-04-06: Fixed a daemon manifest port validation gap found during the Security review sweep. Shared daemon manifest parsing accepted any positive integer `wsPort`, so malformed or tampered `daemon.json` files with out-of-range ports could still be trusted by desktop and CLI discovery. The fix constrains `wsPort` to the TCP port range and adds regression coverage in shared, desktop, and server daemon-manifest readers.
- 2026-04-06: Fixed a stale PID ownership bug found during the Security review sweep. Daemon startup would kill any live PID left in `~/.forge/forge.pid` after a failed `daemon.ping`, even when that PID belonged to an unrelated process. The fix only force-terminates live processes that are plausibly the Forge daemon (trusted matching manifest or Forge daemon command line), keeps stale unrelated PIDs intact, hardens socket pinging to return false cleanly when `forge.sock` is missing, and adds regression coverage for the unrelated-live-PID path.
- 2026-04-06: Fixed release/docs product identity drift found during a Product Identity review sweep. The GitHub release workflow was still building the old `@t3tools/web`/`t3` targets and publishing releases as "T3 Code", and internal docs still pointed at `T3CODE_*`, `~/.t3`, `t3://`, `@t3tools/*`, and `npx t3`. The fix rewires the release workflow to Forge package names and branding, updates the internal docs to Forge identity/runtime terminology, and restores a clean product-facing identity grep for shipped automation and onboarding guidance.
- 2026-04-06: Fixed a singleton discovery regression found during the Spec Compliance review sweep. A duplicate daemon start could kill and replace a healthy responsive daemon when `daemon.json` was missing, malformed, or untrusted, because startup treated the manifest as authoritative after a successful `daemon.ping`. The fix makes socket responsiveness authoritative, keeps duplicate starts in the `already-running` path even without a trusted manifest, validates manifest permissions/socket path before surfacing it, and adds regression coverage for a live daemon whose manifest disappears mid-run.
- 2026-04-06: Fixed daemon WebSocket token drift found during the Integration review sweep. Daemon startup was generating one random `wsToken` for `daemon.json` and a separate random `authToken` for the WebSocket server, so desktop discovery could trust the manifest and still be rejected on connect. The fix threads the actual daemon auth token through `runDaemonModeServer` into `DaemonService`, persists that token into `daemon.json`, and adds regression coverage for both the runtime handoff and persisted manifest.
- 2026-04-06: Fixed a desktop/daemon bootstrap race found during the Integration review sweep. Tray clicks, tray menu activation, and app re-activation could create a renderer window before daemon discovery finished, which let the desktop shell load without a valid daemon WebSocket URL and fall back to `forge://app` as the server origin. The fix gates new desktop window creation on daemon WebSocket readiness, keeps deep links queued until the daemon is ready, and adds focused desktop coverage for the readiness guard.
- 2026-04-06: Fixed dead daemon notification wiring found during the Dead Code review sweep. `NotificationDispatch` was registered in daemon mode but never subscribed to orchestration events, so OS notifications could not fire outside unit tests. The fix adds a daemon-only `NotificationReactor`, starts it from the daemon runtime, filters notifications to top-level sessions and standalone deliberations, and adds focused reactor plus runtime coverage for the live trigger paths.
- 2026-04-06: Fixed daemon manifest parsing drift found during the Code Consistency review sweep. Desktop and CLI/server each had their own `daemon.json` parser and trust checks, which made the acceptance rules diverge and increased the risk of future review regressions. The fix extracts shared manifest parsing/trust helpers into `@forgetools/shared/daemon`, rewires desktop and CLI/server readers to use the shared path, and adds regression coverage for malformed empty-field manifests in shared, desktop, and server tests.
- 2026-04-06: Fixed a daemon manifest trust gap found during the Security review sweep. Desktop and CLI were accepting any readable `daemon.json` contents without validating owner-only permissions or the expected Forge socket path, which could trust a stale or tampered WebSocket token manifest. The fix adds shared daemon-manifest validation, enforces `0600` manifest permissions before token use on supported platforms, verifies `socketPath` matches the expected `~/.forge/forge.sock`, and adds regression coverage in shared, desktop, and server tests.
- 2026-04-06: Fixed a daemon startup readiness race discovered during the Spec Compliance review sweep. The Unix socket could answer `daemon.ping` before the daemon WebSocket listener was actually accepting connections, which let desktop discovery succeed and then race a dead `ws://127.0.0.1:{port}` on first connect. The fix exposes HTTP-listener readiness from `ServerRuntimeStartup`, gates daemon socket RPC execution on that readiness, and adds regression coverage for delayed `daemon.ping` responses until the runtime is ready.
- 2026-04-06: Fixed a JSON-RPC notification handling gap found during the Test Coverage review sweep. The daemon socket transport was incorrectly emitting error responses for notification requests without an `id`, which violates JSON-RPC 2.0 and could confuse fire-and-forget daemon clients. The fix suppresses unknown-method and invalid-params responses for true notifications, adds regression coverage for silent notification failures, and adds a shutdown-order test proving `daemon.stop` responds before the async stop hook runs.
- 2026-04-06: Fixed a missing daemon restart path found during the Spec Compliance review sweep. The CLI exposed `forge daemon start|stop|status`, but the daemon-mode design and auto-update contract rely on `forge daemon restart` to roll the singleton cleanly after version changes. The fix adds a restart command, waits for `daemon.stop` to fully release the socket before relaunch, and adds CLI coverage for the stop-then-launch flow.
- 2026-04-06: Fixed daemon CLI interaction drift found during the Spec Compliance review sweep. The daemon transport still required gate `threadId` + explicit `phaseRunId`, and the `forge` CLI was missing the user-facing `approve`, `reject`, and `intervene` commands called for by the daemon-mode design. The fix teaches `gate.approve`/`gate.reject` to accept `sessionId`, resolves the current pending gate when unambiguous, keeps ambiguous sessions safe by requiring `--phase-run-id`, adds the missing CLI commands, and covers the new transport/CLI behavior with regression tests.
- 2026-04-06: Fixed daemon shutdown drift found during the Integration review sweep. The daemon runtime was starting `DaemonService` with a no-op graceful-shutdown hook, so `daemon.stop` skipped stopping active provider sessions before transport teardown. The fix threads `ProviderService` into daemon runtime startup, stops active sessions during graceful shutdown, drains pending notification work before teardown, and adds runtime coverage for the shutdown path.
